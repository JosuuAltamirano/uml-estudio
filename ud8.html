Entornos de Desarrollo · Examen 11/06/2026

¿Qué es un Diagrama de Casos de Uso?

Son los diagramas más conocidos de UML y representan los diferentes actores externos que interactúan con el sistema de información, y a través de qué funcionalidades (casos de uso o requisitos funcionales) se relacionan con él.

Pregunta clave: "¿Qué puede hacer el sistema desde el punto de vista del usuario?"

Muestran de manera visual las diferentes funciones que puede realizar cada tipo de usuario de un sistema de información.

Finalidades
  • Representar los requisitos funcionales del sistema (qué queremos que haga).
  • Representar los actores que se comunican con el sistema (usuarios u otros sistemas externos).
  • Representar la relación entre los requisitos funcionales y los actores.
  • Guiar el desarrollo del sistema, dándonos un punto de partida.
  • Facilitar la comunicación precisa entre cliente y desarrollador, simplificando que todos participen en elaborar los requisitos.

Actores

Son las entidades externas que interactúan con el sistema.

  • Pueden ser personas (usuarios) o sistemas externos
  • Representan roles, no individuos concretos
  • Se dibujan como figura de muñeco (stickman)
  • Se colocan fuera del rectángulo del sistema
Ejemplos: Cliente, Administrador, Bibliotecario, Sistema de pago, Pasarela bancaria
Casos de Uso

Representan una funcionalidad (unidad de funcionalidad completa) que realiza el sistema.

  • Suelen ser acciones: "Realizar pedido", "Listar productos", "Enviar mensaje"
  • Se representan como una elipse con el nombre dentro
  • Se colocan dentro del rectángulo del sistema
Sistema — Límite del Sistema

Define qué parte pertenece al sistema y qué queda fuera.

  • Se representa como un rectángulo que engloba los casos de uso
  • Ayuda a delimitar el alcance del sistema
  • Los actores quedan fuera del rectángulo
Las 4 Relaciones
1. Asociación

La más básica. Indica que el actor participa en esa funcionalidad. Une un actor con un caso de uso.

Actor ↔ Caso de uso Sin flecha
2. Inclusión  «include»

El caso de uso siempre utiliza otro caso de uso de forma obligatoria. Se usa para reutilizar funcionalidad común.

OBLIGATORIO · siempre se ejecuta Flecha → al incluido
3. Extensión  «extend»

El caso de uso puede ampliar opcionalmente a otro en ciertas condiciones. Comportamientos adicionales o alternativos.

OPCIONAL · solo bajo condición Flecha → al caso base
4. Generalización

Relación de herencia entre actores o casos de uso. El hijo hereda el comportamiento del padre.

Triángulo vacío → apunta al padre Herencia de comportamiento
9 Pasos para Dibujar un Diagrama

3

4

5

6

7

8

Determinar relaciones entre casos de uso e identificar acciones que se repiten.

9

Comprobar la vista general para que todo esté correcto.

Interpretar Diagramas — Ingeniería Inversa

Al igual que obtenemos el diagrama a partir de unos requisitos, es importante saber interpretar los diagramas de uso.

Como desarrolladores, nos llegará el diagrama ya hecho sin explicación y debemos diseminarlo y entender cuál es la lógica detrás de él.
Errores Comunes
  • Confundir actores con personas concretas — (Juan, María) ❌ → Roles (Cliente, Administrador) ✅
  • Diagramas demasiado complejos — muchos casos de uso y relaciones dificultan la lectura. Mejor simples y claros.
  • Mal uso de «include» y «extend»:
    • Usar «extend» cuando es obligatorio → debe ser «include»
    • Usar «include» para cosas opcionales → debe ser «extend»
  • No definir el límite del sistema — siempre dibujar el rectángulo.

Truco: Include vs Extend
«include» = SIEMPRE. Es obligatorio. La funcionalidad siempre necesita la otra.
Ej: "Realizar compra" → siempre → "Validar pago"
«extend» = A VECES. Bajo condición. Solo cuando se cumple algo.
Ej: "Aplicar cupón" → solo si el cliente tiene cupón válido
Errores Comunes — Memoriza
  • ❌ Actores: Juan, María  →  ✅ Roles: Cliente, Admin
  • ❌ «extend» para algo obligatorio  →  ✅ usar «include»
  • ❌ «include» para algo opcional  →  ✅ usar «extend»
  • ❌ Sin rectángulo del sistema  →  ✅ siempre dibujarlo
  • ❌ Diagrama muy complejo  →  ✅ simple y claro
