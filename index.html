¿Qué es UML y para qué sirve?

UML (Unified Modeling Language)

Es un lenguaje gráfico para diseñar software antes de programarlo. No es un lenguaje de programación — sirve para describir la estructura del sistema mediante diagramas.


Su objetivo: que todo el equipo entienda cómo va a ser el software antes de escribir código.

Tipos de diagramas UML

Estructurales — el QUÉ
  • Diagrama de clases ← el más importante
  • Diagrama de objetos
  • Diagrama de componentes
  • Diagrama de paquetes
De comportamiento — el CÓMO
  • Diagrama de casos de uso
  • Diagrama de secuencia
  • Diagrama de actividades
  • Diagrama de estados
El diagrama de clases es el MÁS IMPORTANTE porque muestra la base estructural de la aplicación.

¿Qué es un Diagrama de Clases?

Representa la estructura estática del sistema orientado a objetos. Muestra:

  • Las clases del sistema
  • Sus atributos y métodos
  • Las relaciones entre ellas

La Clase — Estructura

Una clase se dibuja como un rectángulo dividido en 3 zonas:


NombreClase
- atributo1: tipo
- atributo2: tipo
# atributo3: tipo
+ metodo1(): void
+ metodo2(): String
Zona 1 — Nombre
El nombre de la clase. En negrita, con mayúscula inicial.
Zona 2 — Atributos
Los datos/propiedades. Se pone la visibilidad antes.
Zona 3 — Métodos
Las acciones que puede hacer. Misma visibilidad.

Visibilidad

SímboloNombreQuién puede acceder
+PúblicoCualquier clase
-PrivadoSolo la misma clase
#ProtegidoLa clase y sus subclases (herencia)
Los atributos suelen ser privados (-) y los métodos públicos (+). Es la encapsulación.

Formato de atributos y métodos

Atributo: visibilidad nombre: tipo

Ej: - precio: float


Método: visibilidad nombre(parametros): tipoRetorno

Ej: + comprar(v: Videojuego): void

Getters, Setters y Constructores

Los getters, setters y constructores NO se ponen en el diagrama (salvo que tengan relevancia especial).
// Getter
+ getNombre(): String

// Setter
+ setNombre(n: String): void

El patrón es simple:

  • getter = get + NombreAtributo
  • setter = set + NombreAtributo

Pero recuerda: no los pones en el diagrama.

Clase Abstracta

Una clase que no se puede instanciar directamente. Solo sirve de base para otras clases.

Se marca con <<abstract>> o con el nombre en cursiva.


<<abstract>>
Usuario
# nombre: String
# email: String
+ login(): void

Atributos/métodos estáticos

Los atributos o métodos estáticos se representan subrayando su nombre.

Son los que pertenecen a la clase, no a cada objeto.

Relaciones entre Clases

1. Asociación — "se relaciona con"

Nave
────────
1        1
Piloto
  • Línea continua simple entre las dos clases
  • La más común. Representa cualquier relación semántica
  • Puede ser bidireccional (línea) o unidireccional (flecha →)
  • Se añaden las multiplicidades en los extremos

2. Herencia — "es un tipo de"

Perro
───────▷
Animal
  • Línea con flecha de triángulo hueco apuntando a la clase padre
  • Clave: "Perro ES UN Animal"
  • La subclase hereda todos los atributos y métodos del padre
En el examen: si ves que dos clases comparten atributos → piensa en herencia

3. Agregación — "tiene un" (independiente)

Nave
◇───────
Propulsor
  • Rombo vacío (◇) en la clase contenedora
  • La parte puede existir sin el todo
  • Clave: "la Nave TIENE UN Propulsor" pero el Propulsor existe solo
  • Ejemplo: Equipo tiene Jugadores, pero los Jugadores existen sin el equipo

4. Composición — "tiene un" (dependiente)

Frase
◆───────
Palabra
  • Rombo relleno (◆) en la clase contenedora
  • La parte NO puede existir sin el todo
  • Si se destruye el contenedor, se destruyen las partes
  • Ejemplo: Capítulo no existe sin Libro, Habitación no existe sin Casa

5. Dependencia — "usa temporalmente"

GestorPagos
- - - - →
PasarelaPagos
  • Línea discontinua con flecha
  • Una clase usa temporalmente a otra (métodos, parámetros)
  • No es una relación permanente
  • Típico con librerías externas

Multiplicidades — ¿Cuántos?

NotaciónSignificadoEjemplo
1Exactamente unoUn pedido tiene 1 cliente
0..1Cero o unoUn empleado puede tener 0 o 1 jefe
0..* o *Cero o variosUn cliente tiene 0 o más pedidos
1..*Uno o varios (al menos uno)Un módulo tiene 1 o más profesores
N..MDesde N hasta MUn equipo tiene 5..11 jugadores
Las multiplicidades se leen así: Alumno 0..* — matrícula — 1..* Módulo = "Un alumno puede matricularse en 0 o más módulos, y un módulo tiene al menos 1 alumno"

Agregación vs Composición — Truco

Hazte esta pregunta: "¿Si borro el 'todo', las 'partes' siguen existiendo?"


SituaciónRelación
Si cierro el Centro Comercial → los Clientes siguen◇ Agregación
Si destruyo el Libro → los Capítulos desaparecen◆ Composición
Si borro al Usuario → su Biblioteca desaparece◆ Composición
Si borro la Biblioteca → los Videojuegos siguen◇ Agregación

Proceso para hacer un diagrama de clases

Cuando te dan un enunciado en el examen, sigue siempre este orden:

  1. Identificar las clases

    Busca los sustantivos del enunciado. Cada entidad importante = una clase.
    Ejemplo: "usuarios compran videojuegos del catálogo" → Usuario, Videojuego, Catálogo

  2. Definir atributos y métodos de cada clase

    Para cada clase piensa: ¿qué datos tiene? → atributos. ¿qué puede hacer? → métodos.
    Atributos privados (-), métodos públicos (+).

  3. Identificar las relaciones

    Para cada par de clases pregúntate qué relación tienen:
    ¿Una hereda de la otra? ¿Una contiene a la otra? ¿Se usan mutuamente?

  4. Añadir multiplicidades

    En cada relación, ¿cuántos objetos de cada lado pueden participar?
    1:1, 1:0..*, etc.

  5. Conectar y dibujar

    Dibuja las clases y conecta con el tipo de línea correcto según la relación.

Preguntas clave para identificar relaciones

¿A es un tipo de B?

→ Herencia A → B

¿A tiene un B, pero B puede existir solo?

→ Agregación A ◇ B

¿A tiene un B, y B no existe sin A?

→ Composición A ◆ B

¿A y B se relacionan de alguna manera?

→ Asociación A — B

Los 3 pasos del proceso

1. Busca los sustantivos → son las clases

2. Define atributos (datos) y métodos (acciones) para cada clase

3. Identifica las relaciones entre clases y añade multiplicidades

NO poner en el diagrama: getters, setters, constructores.
SÍ poner: métodos relevantes de negocio (+comprar, +valorar, etc.)
Truco herencia: "¿Un X es un Y?" → Si sí, hay herencia.
Truco composición: "¿Si borro A, B también desaparece?" → Si sí, composición.

Test de repaso

Haz clic en la respuesta. Se corrige al momento.

1. ¿Qué tipo de relación usarías si "Perro" deriva de "Animal"?

Correcto. Perro ES UN Animal → Herencia. Línea con triángulo hueco apuntando a Animal.

2. Una "Biblioteca" pertenece a un "UsuarioRegistrado" y no puede existir sin él. ¿Qué relación es?

Correcto. La biblioteca NO existe sin el usuario → Composición (rombo relleno ◆).

3. ¿Qué símbolo de visibilidad se usa para un atributo privado?

Correcto. (-) privado, (+) público, (#) protegido.

4. Un "Catálogo" contiene "Videojuegos", pero los videojuegos pueden existir sin el catálogo. ¿Qué relación es?

Correcto. Los videojuegos pueden existir solos → Agregación (rombo vacío ◇).

5. ¿Qué significa la multiplicidad "1..*"?

Correcto. 1..* = al menos uno, puede haber más. Ej: Un módulo tiene 1 o más profesores.

6. ¿Cuál de estas cosas NO se suele poner en un diagrama de clases?

Correcto. Los getters, setters y constructores NO se ponen salvo que tengan relevancia especial.

7. ¿Cómo se representa la dependencia en UML?

Correcto. La dependencia es una línea discontinua (- - - →). Indica uso temporal de otra clase.

8. Tienes las clases "UsuarioRegistrado" y "Administrador". Ambas tienen nombre, email y contraseña. ¿Qué harías?

Correcto. Cuando dos clases comparten atributos, se crea una clase padre con esos atributos comunes y las otras heredan de ella.

9. ¿Cuál es el primer paso para dibujar un diagrama de clases a partir de un enunciado?

Correcto. Siempre: 1) Clases (sustantivos) → 2) Atributos y métodos → 3) Relaciones.

10. ¿Qué símbolo usa el atributo protegido, accesible desde la clase y sus subclases?

Correcto. (#) protegido = accesible desde la clase y sus subclases (herencia). Muy útil en clases abstractas.
